Function calls

You’ll be using functions a lot, from those available in R packages to ones you write. Most functions perform some action(s) to some object(s). When using functions

  • you can leave out the name of the data argument (generally this is first);
  • write out the names of other arguments;
  • do not write out the default arguments and their values unless you change the values.
# Good
mean(1:20)
ggplot(mtcars, mapping = aes(x = hp, y = mpg)) +
  geom_point()

# Bad
mean(1:20, trim = 0, na.rm = FALSE)
ggplot(data = mtcars, aes(x = hp, y = mpg)) +
  geom_point()

Writing functions

Functions in programming and data science are not unlike the mathematical functions you learned about in previous courses. Functions depend on a set of inputs. You’ll pass values into those inputs and based on the function’s purpose it will generally return a result.

Function syntax

A function is comprised of arguments (formals), a body, and an environment. The first two will be our focus as we use and develop these objects.

Below is the syntax for writing your own function in R.

fcn_name <- function(arg_1, arg_2) {

  # function's body
  # code that does the work goes here

}
  • fcn_name is a name you choose for your function.
  • function is a keyword in R that you will use when you create your function.
  • arg_1, arg_2 are the function’s arguments. You choose how many, their names, and their purpose. Keep the names short and descriptive.
  • Between the { } will be the R code for your function’s body. Generally, this code will manipulate data based on the values input to the arguments and then return a result.

Function call

Run the code below to see this function in action. We’ll always need to load the function into our environment before using it. This is as simple as putting it in an R chunk in your R Markdown document and hitting the green play button.

preview_data <- function(data) {
  data %>% 
    sample_n(size = 5)
}

preview_data(data = mtcars)

Note! Here we have data = mtcars to be explicit. Typically you would make your function call by preview_data(mtcars) as the data argument is typically first.

R automatically returns the result of the last evaluated expression inside a function. Alternatively, you can use return() explicitly. For example, another way to write preview_data() is

preview_data <- function(data) {
  x <- data %>% 
    sample_n(size = 5)
    
  return(x)
}

Best practices

Now that you know the syntax of a function in R and the basics, it’s time to explore some best practices with regards to writing functions. When should you write a function? What should you name your function? Should you use return()?

When should you write a function?

If you copy the same code more than twice, write a function. For example, suppose you have vectors of numeric data you want to standardize. Rather than writing this code

x <- 1:10
y <- c(4, 0, -4)
z <- c(0.01, 0.1, 1, 10, 100, 1000)

(x - mean(x)) / sd(x)
(y - mean(y)) / sd(y)
(z - mean(z)) / sd(z)

write a function with a single data argument.

standardize <- function(data) {
  (data - mean(data)) / sd(data)
}

x <- 1:10
y <- c(4, 0, -4)
z <- c(0.01, 0.1, 1, 10, 100, 1000)


standardize(x)
standardize(y)
standardize(z)

Writing functions will help make your code more readable, and you’ll be less prone to copy-paste mistakes. Also, if you ever need to revise your standardization procedure, you only need to do so in the function’s body. This is much more efficient.

Naming

Functions generally perform some action to a set of data conditional on a set of values passed to the function’s arguments. When naming your function, choose a short and memorable verb. Think about the dplyr functions you know: select(), filter(), group_by(), summarize(). All are short, memorable, and give some insight as to what the function does. Strive to accomplish this when you name your functions.

Consider a set of functions you would create to scrape and save available jobs from an online job board.

# Good
get_jobs()
clean_jobs()
write_jobs()


# Bad
jobs_from_online_job_board()
jobs_cleaned()
saving_my_jobs()

Using return()

A function written in R will automatically return the last expression evaluated. It is recommended that you only use return() for early returns. This will occur when you start writing more complex functions with control flow. For now, rely on R to return the result.

# Good
is_letter <- function(l) {
  l %in% letters
}

# Bad
is_letter <- function(l) {
  return(l %in% letters)
}
